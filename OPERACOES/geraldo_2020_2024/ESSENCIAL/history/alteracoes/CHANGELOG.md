## PS9 — runtime/ trocado por Python embeddable real, sem dependência de Anaconda (2026-06-27)
- **Achado crítico**: o usuário exigiu que a aplicação funcione numa máquina sem Python/VS Code instalado. Ao verificar, `ESSENCIAL/runtime/` (usado desde o PS0) era um `venv` criado a partir do Anaconda local (`pyvenv.cfg` apontava para `C:\Users\...\anaconda3`; `runtime\Lib\` só tinha `site-packages`, sem a biblioteca padrão) — **não era portátil de verdade**. Confirmado na prática: o processo do venv resolvia para `anaconda3\python.exe` em tempo de execução (visto via `Get-CimInstance Win32_Process`).
- Reconstruí `runtime/` com a distribuição **Python 3.12.3 embeddable oficial** (python.org), com pip bootstrapado via `get-pip.py` e as dependências (`requirements.txt`) instaladas dentro dela. `launcher_exe.py` já tinha suporte pronto pra isso (`_encontrar_python()` já procurava `runtime\python.exe` como prioridade 2) — nenhuma mudança de lógica de descoberta foi necessária ali.
- **2º problema encontrado durante a validação**: mesmo com Python embeddable, o `pip install` inicial "vazou" para o `site-packages` do perfil do usuário do Windows (`%APPDATA%\Python\Python312\site-packages`) para 4 pacotes (`packaging`, `six`, `colorama`, `python-dateutil`), porque `import site` (necessário para o pip funcionar) também habilita o user-site por padrão. Corrigido: reinstalação forçada com `PYTHONNOUSERSITE=1`, e essa variável agora é definida antes de toda invocação do runtime (`launcher_exe.py` via `subprocess.Popen(..., env=...)`, `iniciar_sistema.bat`, `carregar_xml.bat`) — garante que o runtime nunca decide silenciosamente usar pacotes de fora dele mesmo.
- **3º problema encontrado durante a validação**: `.bat` com caracteres acentuados/em-dash (`—`) corrompiam ao processar comandos em certos contextos de execução (saída redirecionada) — risco real para confiabilidade numa máquina desconhecida, não só artefato de teste. Removidos todos os caracteres não-ASCII de `iniciar_sistema.bat`, `setup_ambiente.bat`, `build_exe.bat`, `carregar_xml.bat` (substituídos por hífen simples); `chcp 65001` também removido (não é mais necessário sem acentos).
- `launcher/setup_ambiente.bat` reescrito para montar o runtime embeddable (baixa o zip do python.org + `get-pip.py` via PowerShell `Invoke-WebRequest`/`Expand-Archive`, nativo do Windows — sem dependência nova) em vez de `python -m venv`, para que um rebuild futuro não regrida para a versão acoplada ao Anaconda.
- `runtime/` antigo preservado em `ESSENCIAL/runtime_OLD_venv_backup/` (não apagado — rollback possível; remover manualmente depois de confirmar que está tudo certo).
- Validado: processo do novo runtime confirmado rodando como `ESSENCIAL\runtime\python.exe` (não mais `anaconda3\python.exe`); painel Streamlit sem regressão (screenshot, sem erros de console); `carregar_xml.bat` testado nos dois cenários (motor próprio em `geraldo_2020_2024/` e motor emprestado em `_MODELO_OPERACAO/`), ambos classificando XML corretamente com o runtime novo e sem corrupção de `.bat`. Artefatos de teste removidos depois.
