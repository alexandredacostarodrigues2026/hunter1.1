# Atualização 2026-07-06

## Contexto do Sistema (recapitulação + termos novos do dia)

Ver `memoria/2026-07-04.md` para o glossário completo do projeto (HUNTER 1.1, NF-e, SPED/EFD, `C100`/`C170`/`0200`/`0190`, ET/EP, `IND_OPER`/`IND_EMIT`, "Auditada", `dtype=str`/Regra R07, DuckDB) e `memoria/2026-07-05.md` para CFOP, situação da NF-e, segregação, **BC1** (Base Comparativa 1 — dataset `sped_entradas_terceiros`, lado SPED) e **BC2** (Base Comparativa 2 — dataset `nfe_bc2`, lado XML, itens de Emissão de Terceiros).

Termos novos de hoje:

- **Matching (Etapa 1)**: a etapa do sistema que efetivamente cruza a BC2 (o que o fornecedor emitiu, segundo o XML) com a BC1 (o que o cliente auditado declarou ao Fisco, segundo o SPED), item a item, para descobrir se cada compra recebida foi corretamente escriturada.
- **BC3**: o resultado do Matching — uma tabela com uma linha por item da BC2, indicando se e como ele foi encontrado na BC1 (e trazendo a descrição/código do item conforme registrado na declaração, quando encontrado).
- **GTIN/EAN**: o código de barras global e padronizado de um produto (Global Trade Item Number / European Article Number) — em teoria, o mesmo produto tem o mesmo GTIN em qualquer nota fiscal de qualquer fornecedor, então é um identificador mais confiável que o código interno de cada empresa.
- **Levenshtein / similaridade de texto (fuzzy matching)**: técnica que compara duas strings e calcula um "score" (0 a 1) de o quanto elas são parecidas, mesmo que não sejam idênticas — usada aqui para comparar a descrição do produto no XML com a descrição do produto na declaração, já que fornecedor e declarante frequentemente escrevem o nome do mesmo produto de formas ligeiramente diferentes (abreviações, ordem das palavras, etc.).

## Resumo Executivo

Implementado o motor de Matching (Etapa 1): cruza a BC2 (XML) com a BC1 (SPED) para produzir a BC3. Essa é uma mudança de premissa importante em relação ao que se cogitava antes: a ideia original era usar `CHV_NFE + NUM_ITEM` (a chave de acesso da nota + o número sequencial do item) como elo entre as duas bases — mas essa premissa foi **abandonada** porque não há garantia nenhuma de que o item nº 3 no XML do fornecedor seja o mesmo item nº 3 na escrituração do declarante (cada lado pode ordenar/numerar os itens de forma diferente). A nova abordagem usa, em ordem de prioridade: valor financeiro exato, depois código de barras (GTIN) exato, depois similaridade de texto na descrição do produto.

## Funcionalidades Implementadas

- **`scoring.py`** (só existia como esqueleto vazio até hoje): duas funções usando a biblioteca `rapidfuzz` (já estava nas dependências do projeto, usada no cálculo de similaridade de texto). `similaridade(a, b)` devolve um número de 0 a 1 dizendo o quanto duas strings são parecidas. `melhor_similaridade(referencia, candidatos)` compara uma string contra uma lista de candidatas e devolve a mais parecida.
- **`matching.py`** (também só existia como esqueleto vazio): a função principal `executar_matching()` faz o cruzamento em duas etapas (com fallback):
  1. **Match Principal** (`_match_principal`): tenta casar cada item da BC2 com um item da BC1 que tenha a **mesma Chave de Acesso (`CHV_NFE`) e o mesmo Valor do Item (`VL_ITEM`)**, arredondado a 2 casas decimais para evitar diferenças de centavos por arredondamento. É uma correspondência 1-para-1: um item da BC1 só pode ser usado para casar com um único item da BC2 (evita que dois itens do XML "roubem" o mesmo item da declaração).
  2. **Match Secundário** (`_match_secundario`): para os itens que sobraram do passo 1, tenta, **só entre itens da mesma nota (mesma `CHV_NFE`)** que ainda não foram usados: primeiro um match exato pelo código de barras (GTIN/EAN); se não achar, calcula a similaridade de texto entre a descrição do produto no XML e a descrição registrada na declaração (registro `0200` do SPED) — se o score for maior que 0,85 (85% de parecença), considera como match sugerido.
  3. Para o que sobrar depois dos dois passos, o sistema agora distingue **dois motivos diferentes** de não-match (isso foi um ajuste feito no meio do dia, depois de uma pergunta do usuário): `'nd'` (não declarado) quando a chave de acesso inteira nem aparece na BC1 — ou seja, a nota de compra não foi encontrada na declaração de jeito nenhum, o que pode indicar uma compra não declarada; e `'nm'` (não match) quando a nota existe na BC1, mas esse item específico não bateu por nenhum dos três critérios — a nota foi declarada, só não foi possível confirmar qual item específico corresponde.
- Nova função `loader.persistir_bc3()`: roda o `matching.executar_matching()` e grava o resultado na tabela `bc3` do DuckDB. Usa "lazy import" (`import matching` só dentro da função, não no topo do arquivo) para evitar um problema de "importação circular" — como `matching.py` já importa `loader.py` (para ler a BC2 e a BC1), se `loader.py` importasse `matching.py` logo no topo também, os dois arquivos ficariam esperando um pelo outro para carregar e o programa não funcionaria; importar só na hora de usar resolve isso.
- `loader.bc3_ja_gerada()` / `loader.consultar_bc3()`: seguem o mesmo padrão de "verificar o que já está persistido no banco antes de reprocessar" usado em todas as outras telas do sistema até agora.

## Decisões Técnicas

- **Só a camada de dados foi implementada hoje** (funções em `loader.py`/`matching.py`/`scoring.py`, testadas via linha de comando) — sem botão nem tela na interface (`interface.py`/`main.py`) ainda. Isso segue o mesmo padrão já usado para a BC1 e a BC2: primeiro a lógica de dados é implantada e validada, e o botão/tela na interface vem depois, numa mensagem separada, se o usuário pedir.
- O valor (`VL_ITEM`) é convertido para número (`float`) só internamente, na hora de comparar — a coluna continua sendo tratada como texto (`dtype=str`) no restante do sistema, seguindo a Regra Operacional R07.
- `ID_UNICO` (a coluna de identificação única criada em 2026-07-05) continua existindo em ambas as bases, mas **deixou de ser usada como critério de cruzamento** — agora serve só para rastreabilidade (poder localizar de volta a linha original no banco, se precisar).

## Soluções Aplicadas

- Validado contra dados reais da operação `geraldo_2020_2024`: dos 16.408 itens da BC2 (itens de Emissão de Terceiros), 2.529 casaram pelo valor exato (Match Principal), 1.920 casaram pela similaridade de texto (nenhum casou por código de barras nesta operação — os dados de GTIN podem estar ausentes ou divergentes aqui), 846 ficaram como "não declarado" (nota toda ausente da BC1) e 11.113 como "não match" (nota existe, item não confirmado). A soma bate exatamente com o total (2.529+1.920+846+11.113 = 16.408) — nenhum item ficou de fora da contagem.
- Conferência manual de amostras: os matches por similaridade de texto (ex.: "IOGURTE NATURAL PARCIALMENTE DESNATADO ISIS 150 G" com "IOG NATURAL PARCIALMENTE DESNATADO ISIS 150GR", score 0,91) fazem sentido — claramente o mesmo produto escrito de forma abreviada.
- Testado `persistir_bc3()` de ponta a ponta: gravou 16.408 linhas na tabela `bc3` sem erros.

## Problemas Encontrados

- O processamento completo leva cerca de 45 segundos para os ~16 mil itens desta operação (o cálculo de similaridade de texto item a item é a parte mais lenta). Isso é aceitável para uma ação sob demanda (com uma barra de carregamento, por exemplo), mas pode ficar mais lento se uma operação futura tiver muito mais itens — vale otimizar se isso incomodar no futuro (por exemplo, comparando só contra os itens da mesma nota, o que já é feito, mas poderia se beneficiar de uma biblioteca de comparação em lote em vez de calcular um por um).
