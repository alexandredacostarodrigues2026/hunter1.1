# Atualização 2026-07-04

## Contexto do Sistema (para quem está lendo sem ter acompanhado o desenvolvimento)

O **HUNTER 1.1** (nome do produto: "Equalizador de Produtos") é um sistema de auditoria fiscal. Sua função é comparar duas fontes de informação sobre as mesmas operações comerciais de uma empresa (a "operação auditada") para encontrar divergências:

- **Lado XML**: as Notas Fiscais Eletrônicas (**NF-e**) de fato emitidas/recebidas, extraídas em arquivos `.txt` a partir dos XMLs originais. Cada NF-e tem uma **Chave de Acesso** (`CHV_NFE`, sempre 44 dígitos) que a identifica de forma única no país inteiro.
- **Lado Declaração**: o **SPED** (Sistema Público de Escrituração Digital), especificamente a **EFD ICMS/IPI** — o arquivo mensal que a empresa é obrigada a entregar ao Fisco resumindo tudo o que ela declarou ter comprado/vendido/estocado. O layout desse arquivo é público (Guia Prático da Escrituração Fiscal Digital) e organizado em "registros" com códigos como `C100`, `C170`, `0200`, `0190`, `H010`.

O objetivo do cruzamento (chamado no projeto de "Etapa 1" / "Matching") é pegar os itens comprados que aparecem no XML e nos itens declarados no SPED e verificar se batem — se uma nota foi recebida mas não declarada (ou vice-versa), isso é uma divergência fiscal relevante para auditoria.

Cada cliente auditado é chamado de **"operação"** no projeto (ex.: `geraldo_2020_2024`, `PB2`) e tem sua própria pasta com o mesmo motor de código (`ESSENCIAL/app/loader.py` + `interface.py` + `main.py`, interface feita em **Streamlit**) e seu próprio banco de dados local (**DuckDB**, um arquivo `hunter.duckdb` por operação) onde os dados já processados ficam gravados para consulta rápida, sem precisar reler os arquivos brutos toda hora.

Alguns termos técnicos usados a partir daqui:

- **Auditada**: a empresa (identificada por um CNPJ específico) cujas operações comerciais e registros fiscais são o alvo da análise. Cada empresa auditada é tratada no sistema como uma **"operação"** individual, com sua própria pasta de arquivos e seu próprio banco de dados DuckDB — ver definição de "operação" logo abaixo.
- **ET (Emissão de Terceiros)**: as notas fiscais eletrônicas em que a empresa auditada figura como **destinatária** da mercadoria ou serviço — ou seja, representam as **compras** que a auditada realizou junto aos seus fornecedores. Do lado XML, ficam na pasta `ET/`. Do lado da declaração (SPED), esse subconjunto é extraído pelos filtros `IND_OPER = 0` (Entrada) + `IND_EMIT = 1` (Terceiros).
- **EP (Emissão Própria)**: os documentos fiscais em que a própria empresa auditada é a **emitente**. Do lado XML, ficam organizados na pasta `EP/`. Para fins de escrituração no SPED, esses registros são identificados pelo campo `IND_EMIT = 0`.
- **`C100`**: registro do SPED que representa o cabeçalho de um documento fiscal inteiro (uma nota).
- **`C170`**: registro do SPED que representa um item/produto dentro de uma nota — várias linhas `C170` pertencem a um único `C100` (relação pai-filho).
- **`0200`/`0190`**: registros de *cadastro* do SPED (não são movimento, são tabelas de referência) — `0200` é o cadastro de produtos do declarante (código interno, descrição, NCM, código de barras); `0190` é o cadastro das unidades de medida usadas (ex.: "UN", "CX", "KG").
- **`IND_OPER`**: campo do `C100` que diz se o documento é de Entrada (`0`) ou Saída (`1`).
- **`IND_EMIT`**: campo do `C100` que diz quem emitiu o documento sob a ótica de quem declarou o SPED — `0` = emissão própria, `1` = emissão de terceiros. A combinação `IND_OPER=0` + `IND_EMIT=1` = "documento de entrada emitido por terceiros", ou seja, uma compra normal recebida de um fornecedor — esse é o subconjunto mais importante para cruzar com o XML.
- **`dtype=str` / "Regra Operacional R07"**: convenção do projeto (nome dado pelo próprio usuário) de nunca deixar o Pandas (biblioteca Python de dados) inferir tipo numérico em campos que são, na prática, códigos — como `CHV_NFE`, `COD_ITEM`, `NUM_ITEM`. Se isso acontecer, um código como `"00001234"` viraria o número `1234`, perdendo os zeros à esquerda, e o cruzamento por igualdade de texto quebraria silenciosamente.

Com esse pano de fundo, seguem as atualizações do dia.

## Resumo Executivo

Implementadas as **"chaves de entrada de emissão de terceiros"** — o subconjunto do SPED que representa compras recebidas de fornecedores (`IND_OPER=0` + `IND_EMIT=1`), já enriquecido com a descrição do produto e da unidade de medida — no `loader.py` (o motor de carga de dados, compartilhado pelas operações `geraldo_2020_2024`, `PB2` e `_MODELO_OPERACAO`, um template vazio usado para criar operações novas). Também foi criado o `DICIONARIO DE CAMPOS.txt`, um arquivo de referência na raiz do projeto que traduz os nomes técnicos de colunas (ex.: `fatonfe_infnfe_emit_cnpj`) para nomes legíveis em português (ex.: "Número do CNPJ do emitente").

## Funcionalidades Implementadas

- **Leitura do registro `0190`** (cadastro de unidades de medida) do SPED — antes o sistema só lia `0200`, `C100`/`C170` e `H010` (estoque); agora também lê `0190`, necessário para traduzir o código de unidade (ex.: "UN") em descrição (ex.: "Unidade").
- `_parse_itens_c170_com_c100` (a função que junta cada item `C170` com os dados do `C100`-pai) agora também copia o campo `IND_EMIT` do `C100` para cada item — antes só copiava `IND_OPER`, `NUM_DOC`, `CHV_NFE`, `DT_DOC`, `COD_MOD`. Sem o `IND_EMIT`, não seria possível distinguir "comprei de um fornecedor" (`IND_EMIT=1`) de "emiti uma nota de entrada para mim mesmo" (`IND_EMIT=0`, ex.: nota de entrada avulsa para registrar recebimento de produtor rural).
- Nova função `_enriquecer_itens_com_cadastro()`: pega os itens de compra (`C170`) e junta com o cadastro de produto (`0200`, ligação pelo código interno `COD_ITEM`) e com a descrição da unidade de medida (`0190`, ligação pelo código `UNID`) — assim o item deixa de ser só um código numérico e passa a ter descrição legível.
- Nova função `load_declaracao_entradas_terceiros()`: o dataset final — todos os itens comprados de terceiros (`IND_OPER=0` + `IND_EMIT=1`), já com descrição de produto e unidade anexadas. Esse é o dataset que futuramente vai alimentar o cruzamento com o XML.
- `persistir_sped()` (função que grava os dados processados no banco DuckDB) passou a gravar também as tabelas `sped_unidades` (cadastro `0190`) e `sped_entradas_terceiros` (o dataset acima), além das tabelas que já existiam.
- Criado o dicionário geral de campos: `AI_HUNTER1.1/DICIONARIO DE CAMPOS.txt`, um arquivo texto simples (`campo_tecnico;nome_amigavel`) que mapeia os nomes de coluna técnicos do sistema (prefixos `fatonfe_...`/`fatoitemnfe_...`, herdados de uma extração antiga em formato Qlik) para nomes legíveis em português. Foi iniciado a partir dos 69 campos encontrados em `DADOS BRUTOS/PB_DISTRIBUIDORA/NOME/` (dados de uma distribuidora específica) e, no mesmo dia, generalizado para servir de referência para qualquer fonte de dados do sistema, não só aquela distribuidora.

## Atualização (2ª parte do dia) — Botão no front-end

### Resumo

Adicionado um botão dedicado na interface (Streamlit), que só aparece depois que os dados já foram carregados, para gerar sob demanda as "Chaves de entrada de emissão de terceiros" — em vez de essa geração acontecer automaticamente escondida dentro da carga geral de dados, como estava até então.

### Funcionalidades Implementadas

- `loader.gerar_entradas_terceiros()`: chama `load_declaracao_entradas_terceiros()` (descrita acima) e grava o resultado isoladamente na tabela `sped_entradas_terceiros` do banco — sem precisar reprocessar o XML nem as demais tabelas do SPED, o que torna essa ação rápida mesmo em operações grandes.
- `interface.render_entradas_terceiros()`: cria o botão "Gerar chaves de entrada de emissão de terceiros" na tela; ao clicar, mostra uma mensagem de sucesso com a contagem de registros gerados e uma prévia da tabela (`CHV_NFE`, `COD_ITEM`, `DESCR_ITEM`, `UNID`, `DESCR_UNID` etc.).
- `main.py` (o arquivo que monta a tela inteira do sistema): passou a chamar esse novo botão logo depois da seção "Entidade auditada", e só dentro do bloco que só aparece após a carga de dados já ter acontecido.

### Correções Realizadas

- Como a geração de `sped_entradas_terceiros` deixou de acontecer automaticamente dentro de `persistir_sped()` (função de gravação do SPED no banco), essa função passou a notificar o progresso 4 vezes em vez de 5 (uma vez por tabela: itens, produtos, unidades, estoque). Isso corrigiu de quebra um bug que essa mudança teria introduzido: a barra de progresso na tela ainda esperava 3 avisos (`n_passos=3`, um número fixo no código da interface), e se não fosse ajustada para 4, a barra ultrapassaria 100% e o Streamlit lançaria um erro. Corrigido para `n_passos=4`.

### Problemas Encontrados / Soluções Aplicadas

- Verificação em um navegador de verdade (usando a ferramenta Playwright, que abre um Chrome sem interface gráfica e simula cliques como um usuário faria), rodando o sistema Streamlit de fato com o Python portátil do projeto, na operação `geraldo_2020_2024`: o fluxo completo (carregar dados → as 4 barras de progresso do SPED completando sem estourar → o botão novo aparecendo → clicar nele e gerar as 16.100 linhas com prévia de dados reais) funcionou sem nenhum erro na tela nem no console do navegador (só um aviso inofensivo do próprio Streamlit tentando enviar telemetria para a internet, que falha porque não há acesso).

## Atualização (3ª parte do dia) — Detecção de carga já persistida

### Resumo

O sistema guardava a informação "os dados já foram carregados" só na memória temporária da sessão do navegador (`st.session_state`, um recurso do Streamlit que reseta toda vez que se abre uma aba nova ou se reinicia o programa). Isso significava que, mesmo com os dados já gravados no banco `hunter.duckdb` de uma sessão anterior, toda vez que a tela era reaberta ela "esquecia" e pedia para clicar em "Carregar dados" de novo — o que, se clicado, reprocessaria tudo desnecessariamente. Corrigido: agora, ao abrir a tela, o sistema primeiro consulta o próprio banco de dados para saber se já existe uma carga válida, em vez de assumir que não existe.

### Funcionalidades Implementadas

- `loader.dados_ja_carregados()`: nova função que abre o banco `hunter.duckdb` da operação e verifica se as tabelas `nfe_entradas`/`nfe_saidas` (dados de NF-e) e `sped_itens` (dados do SPED) já existem e têm linhas — sem reprocessar nenhum arquivo, só uma consulta rápida ao banco.
- `main.py`: no início da tela, se a informação "dados carregados" ainda não estiver definida para aquela sessão do navegador (ou seja, é a primeira vez que a tela está sendo desenhada nessa aba), o sistema chama `loader.dados_ja_carregados()` para descobrir o valor correto, em vez de simplesmente assumir que não há dados.

### Correções Realizadas

- A tela não força mais uma carga nova toda vez que é reaberta, se os dados já estiverem persistidos no banco — mostra direto a mensagem "✅ Dados carregados" com um botão opcional "Carregar novamente" (para quando o usuário realmente quiser reprocessar tudo, por exemplo depois de adicionar XMLs novos).

### Soluções Aplicadas

- Validado em um navegador de verdade (Playwright): foi aberta uma instância nova do Streamlit (um processo do zero, nunca visto antes, simulando alguém abrindo o sistema pela primeira vez depois de um reinício) sobre o banco que já estava populado da operação `geraldo_2020_2024` — a tela abriu direto mostrando "✅ Dados carregados", a seção de entidade auditada e a seção de chaves de entrada de terceiros, tudo sem precisar clicar em nada.
