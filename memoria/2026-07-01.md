## Persistência em DuckDB (sessão tarde 2026-07-01)

### Funcionalidades Implementadas

- `loader.persistir_banco(callback=None) -> dict`: lê todos os dados brutos (NF-e .txt + SPED) usando os parsers já existentes e persiste em DuckDB (`ESSENCIAL/banco/hunter.duckdb`). Substituição completa a cada chamada (idempotente). 5 tabelas:
  - `nfe_entradas` — NF-e de entrada (tpnf=0, ET/*.txt)
  - `nfe_saidas` — NF-e de saída (tpnf=1, EP/*.txt)
  - `sped_itens` — registros C100+C170 do SPED
  - `sped_produtos` — cadastro de produtos 0200 do SPED
  - `sped_estoque` — inventário H010 do SPED
- `interface.py`: após a carga de XML, bloco `st.status()` exibe progresso por tabela e total de registros gravados.
- Bloco `__main__` do loader: inclui o passo DuckDB no fluxo CLI com callback de impressão linha a linha.
- `_MODELO_OPERACAO/ESSENCIAL/app/` sincronizado (loader.py + interface.py).

### Validação

- Testado via runtime embeddable com mock de streamlit: 5 tabelas criadas, 155.791 registros no total (nfe_entradas=734, nfe_saidas=75.933, sped_itens=16.100, sped_produtos=37.424, sped_estoque=25.600).
- Banco gerado: `ESSENCIAL/banco/hunter.duckdb` — 12,5 MB (DuckDB colunar, eficiente).
- `hunter.duckdb` já estava em `.gitignore` (`**/banco/*.duckdb`) — correto, contém dados fiscais.

### Decisões Técnicas

- **DuckDB 1.5.4 (já instalado)** escolhido sobre SQLite para cruzamento: colunar, JOIN rápido em centenas de milhares de linhas, sem servidor.
- **Substituição completa** a cada carga (não incremental): garante consistência; arquivos .txt em ET/EP são a fonte de verdade.
- **Função separada de `carregar_operacao()`**: `persistir_banco()` pode ser chamada independentemente (útil para reprocessar sem reclassificar XML).

## Interface e UX (sessão tarde/noite 2026-07-01)

### Funcionalidades Implementadas

- **Botão "Carregar novamente"**: quando já carregado e sem XML pendentes, o botão muda de label para "Carregar novamente" (com tooltip). Permite reprocessar a base sem reiniciar a sessão — útil quando chegam novos arquivos .txt ou SPED.
- **3 barras de progresso independentes**: a carga passou a ter fases visuais separadas:
  - Barra 1 — Classificação de XML (só aparece se houver pendentes, avança arquivo a arquivo)
  - Barra 2 — NF-e: `nfe_entradas` + `nfe_saidas` (2 passos)
  - Barra 3 — SPED: `sped_itens` + `sped_produtos` + `sped_estoque` (3 passos)
  - Numeração adapta: sem XML pendentes, as barras viram "1. NF-e" e "2. SPED"
  - Delay de 0,25s por passo para visibilidade mesmo em cargas rápidas
- **`persistir_nfe()` e `persistir_sped()`**: `persistir_banco()` separado em duas funções independentes para suportar as barras distintas. `persistir_banco()` mantido como wrapper para o CLI.
- **`_barra_progresso()` em interface.py**: helper reutilizável que recebe título, número de passos e função de persistência — evita duplicação de código entre as fases NF-e e SPED.
- **Detecção de operação ativa em `iniciar_sistema.bat`**: ao abrir qualquer bat, verifica se a porta 8600 está ocupada. Se sim, pergunta "Deseja encerrar a operacao atual e abrir esta? (S/N)". S encerra todos os processos na porta e sobe a nova operação; N sai sem alterar nada.
- **`set "HUNTER_OPERACAO_DIR="` em `iniciar_sistema.bat`**: limpa variável herdada do ambiente, evitando que sessão de dev anterior faça PB2 carregar dados de geraldo.

### Correções

- PB2 estava com código antigo (copiado antes das barras de progresso) — atualizado com sync explícito de todos os `.py` de `geraldo/ESSENCIAL/app/` para `PB2/ESSENCIAL/app/`.
- Servidor de teste da sessão de dev ficou na porta 8600 e interceptou a abertura do PB2 — raiz do problema identificada e corrigida no bat.

### Decisões Técnicas

- **Regra de sync de operações**: toda alteração em `ESSENCIAL/app/` de qualquer operação deve ser propagada para geraldo, PB2 e `_MODELO_OPERACAO` antes do commit. Registrado em memória persistente (`feedback_sync_operacoes.md`).
- **Operações de teste ativas**: geraldo_2020_2024 e PB2 são as duas operações em uso simultâneo para validação. Nunca mais subir servidor de teste em PowerShell — testes via bat das próprias operações.
- **`taskkill /F /PID` por porta**: ao encerrar operação ativa, o bat mata todos os PIDs com conexão na porta 8600 (não só o processo pai), garantindo que Streamlit e workers sejam encerrados.
