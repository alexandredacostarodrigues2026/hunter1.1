# Atualização (2ª parte do dia) — Matching Tipo 3

## Resumo Executivo

Implementado o terceiro nível de cruzamento do Matching (Tipo 3, `matching.py`): um dicionário de aprendizado histórico, construído só a partir dos matches já confirmados de Tipo 1/Tipo 2, recupera itens que ficaram como `nd` (não declarado) ou `nm` (sem match) com base em CNPJ do emitente + código do produto no XML + ano de emissão — usando o CNPJ do emitente que passou a estar disponível na BC1 desde a 1ª parte do dia.

## Funcionalidades Implementadas

- `matching._normalizar_codigo()`: normaliza `COD_ITEM` (strip + lstrip de zeros) para comparação, mesma lógica de `_normalizar_gtin` aplicada a código de item.
- `matching._extrair_ano_emissao()`: extrai o ano (dígitos 3-4 da CHV_NFE — campo "AA" de AAMM na chave de acesso da NF-e).
- `matching._chave_aprendizado()` / `_montar_dicionario_aprendizado()`: monta o dicionário de aprendizado (CNPJ_EMITENTE + COD_ITEM XML normalizado + ANO_EMISSAO -> COD_ITEM_DECLARACAO/DESCR_ITEM_DECLARACAO) só a partir das linhas já com `MATCH_TIPO` em `TIPO_1`/`TIPO_2`. Em chave duplicada, prevalece a primeira ocorrência.
- `matching._match_tipo3()`: aplica o dicionário aos itens `ND`/`NM`; ao encontrar correspondência, muda o status para `TIPO_3` e preenche `COD_ITEM_DECLARACAO`/`DESCR_ITEM_DECLARACAO` com o par histórico — inclusive para `ND` (nota nem declarada, mas o padrão de escrituração do fornecedor já é conhecido). Sem correspondência, mantém `ND`/`NM`.
- `executar_matching()` agora chama `_match_tipo3()` após Tipo 1/Tipo 2, e o `meta` retornado ganhou a chave `match_tipo3`.
- `loader.consultar_totais_bc3()`: dicionário de totais ganhou `"TIPO_3": 0`.
- `interface.render_bc3()`: painel agora mostra 5 KPIs (Tipo 1/Tipo 2/Tipo 3/ND/NM) em vez de 4; caption atualizado explicando o critério do Tipo 3.

## Decisões Técnicas

- Dicionário de aprendizado é reconstruído a cada execução do Matching (não persistido à parte) — sempre reflete o estado atual de Tipo 1/Tipo 2 daquela rodada.
- Aplica-se tanto a `ND` quanto a `NM` (requisito explícito do usuário, que revisou o escopo inicial que só cobria `NM`) — uma nota nem declarada pode ainda assim ter o produto "reconhecido" por histórico do mesmo fornecedor/código/ano.
- Normalização de código segue a mesma convenção já usada para GTIN (R07): remove espaços e zeros à esquerda antes de comparar.

## Alterações de Banco de Dados

- Nenhuma alteração de schema na BC3 — Tipo 3 só reescreve `MATCH_TIPO`/`COD_ITEM_DECLARACAO`/`DESCR_ITEM_DECLARACAO` de linhas que já existiam como `ND`/`NM`.

## Resumo Executivo

Implementado o quarto nível de cruzamento do Matching (Tipo 4, `matching.py`): para os itens que ainda sobraram `nd`/`nm` após Tipo 1/2/3, restringe-se às notas "íntegras" (mesma contagem de itens **e** mesmo somatório de `VL_ITEM` entre XML e SPED) e casa por similaridade de descrição > 50%, sem exigir GTIN nem valor por item.

# Atualização (4ª parte do dia) — Correção crítica: vírgula decimal do SPED

## Resumo Executivo

Usuário questionou por que "VINAGRE DE ALCOOL" ficou como `nm` mesmo existindo na declaração com o mesmo valor. Investigação revelou um bug real e sistêmico: o campo `VL_ITEM` do lado SPED (BC1) é gravado pelo EFD no formato brasileiro com **vírgula** decimal (ex.: `"33,6"`), enquanto o XML (BC2) sempre usa ponto (`"33.6"`). `pd.to_numeric()` sem tratamento descarta como `NaN` qualquer valor com vírgula — e `NaN` nunca é igual a nada — quebrando silenciosamente o casamento por Valor Total (Tipo 2) e a soma de integridade de nota (Tipo 4) para a maioria dos itens com centavos.

## Problemas Encontrados

- **82,4% das linhas da BC1** (13.261 de 16.100) têm `VL_ITEM` com vírgula decimal e viravam `NaN` silenciosamente ao converter para numérico em `matching.py` — bug pré-existente desde a criação do Tipo 2 (2026-07-06), só evidenciado agora que o usuário comparou visualmente XML x declaração para uma nota específica.
- O lado XML (BC2) nunca usa vírgula (0 de 16.408 linhas) — o problema era exclusivo do lado SPED.

## Correções Realizadas

- Novo `matching._valor_numerico()`: converte `VL_ITEM` para numérico substituindo vírgula por ponto antes do `pd.to_numeric()`. Usado em `_match_tipo2_por_nota()` (casamento por Valor Total) e `_integridade_por_nota()` (soma de integridade de nota, Tipo 4) — os dois pontos do código que comparavam `VL_ITEM` numericamente.

## Alterações de Arquitetura

- Nenhuma mudança estrutural — só corrige a conversão de tipo já existente.

## Arquivos Modificados

- `OPERACOES/geraldo_2020_2024/ESSENCIAL/app/matching.py`

## Decisões Técnicas

- Correção feita na camada de comparação (`matching.py`), não no `loader.py` — `VL_ITEM` continua armazenado na BC1/DuckDB exatamente como vem do arquivo SPED (vírgula), preservando o dado original para exibição/exportação; só a conversão numérica interna ao Matching foi corrigida.
